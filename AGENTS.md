# AGENTS.md

## 목적

이 문서는 `C:\Users\wkdwo\workspaces\codex\cardnews` 프로젝트의 에이전트 운영 규칙을 한 파일에 모아둔 기준서다.

결과물은 사용자가 요청하는 주제/섹터를 바탕으로 만든 한국어 카드뉴스다. 기본 독자는 비전문가, 1인 사업가, 프리랜서, 작은 회사 대표이며, 최종 산출물은 HTML, 인스타 업로드용 PNG 6장, 가능하면 PDF까지 포함한다.

## 기본 원칙

- `team-lead`는 총괄 데스크다.
- `team-lead`는 직접 리서치, 본문 작성, 이미지 제작을 하지 않고 담당 에이전트에 나눠 맡긴다.
- 단, 최종 산출물이 흔들리면 `team-lead`가 방향을 재정의하고 다음 단계 지시를 내린다.
- 사실이 불확실하면 지어내지 않고 `확인 필요`로 표시한다.
- 모든 작업물은 가능한 한 `output/` 아래에 저장한다.
- 에이전트 보고는 결론 먼저, 근거 나중 순서로 한다.
- 보고 형식은 핵심 3줄 + 다음 단계 1줄을 기본으로 한다.

## 권장 워크플로

최신 권장 흐름은 다음과 같다.

```text
research
→ analysis
→ writing-outline
→ design
→ writing + image
→ packaging
→ design-review
```

핵심 원칙은 이렇다.

- 글의 뼈대는 디자인보다 먼저 잡는다.
- 최종 문구는 디자인 레이아웃이 나온 뒤에 쓴다.
- 이미지는 디자인 브리프가 나온 뒤에 만든다.
- 패키징은 창작이 아니라 구현과 출력, QA를 맡는다.
- 패키징 이후에는 별도 `design-review`가 실제 PNG/PDF 기준으로 메시지 전달력과 디자인 완성도를 검수한다.

## Phase 게이트

### Phase 1: Research

담당: `research`

역할:
- 사용자가 요청한 주제/섹터를 기준으로 국내외 자료를 폭넓게 검색한다.
- 특정 분야의 시장 흐름, 주요 플레이어, 소비자 관심사, 최근 이슈, 실제 사례, 논쟁점, 수치 자료를 함께 수집한다.
- 국내 자료와 해외 자료를 모두 확인하고, 서로 다른 관점이나 지역 차이가 있으면 구분한다.
- 출처의 신뢰도를 확인하고, 원문 출처와 2차 해설을 구분한다.
- 주요 주장, 숫자, 날짜, 인용 가능한 사실은 리서치 단계에서 1차 사실검증을 수행한다.
- 카드뉴스 독자에게 왜 중요한지, 어떤 관점으로 보면 흥미로운지 판단한다.
- 검색 범위가 너무 넓으면 하위 섹터를 나눠 후보군을 만든다.

출력:
- 주제/섹터 정의
- 국내 리서치 요약
- 해외 리서치 요약
- 핵심 후보 아이템 8~12개
- 각 아이템의 한 줄 요약
- 출처 링크와 출처 유형
- 확인된 사실과 확인이 필요한 주장
- 서로 충돌하는 정보 또는 해석 차이
- 카드뉴스 독자 관점의 relevance
- 카드뉴스로 만들었을 때의 흥미도
- 추가 확인이 필요한 키워드
- `확인 필요` 항목

금지:
- 카드뉴스 문구를 완성하지 않는다.
- 디자인을 정하지 않는다.
- 과장된 효과나 보장 표현을 만들지 않는다.

### Phase 2: Analysis

담당: `analysis`

역할:
- `research` 결과를 검토하고 사용자가 요청한 주제/섹터에서 카드뉴스로 만들 만한 각도를 결정한다.
- 리서치의 사실검증 상태를 재검토하고, 위험한 주장·과장된 표현·출처가 약한 내용을 걸러낸다.
- 국내외 자료의 차이, 시장 맥락, 독자 관심사, 시의성을 종합해 핵심 메시지를 정한다.
- 단순 요약이 아니라 “이 주제를 어떤 관점으로 보여줘야 사람들이 관심을 갖는지”를 결정한다.
- 너무 넓은 주제는 하나의 강한 프레임으로 좁히고, 너무 좁은 주제는 독자가 이해할 수 있는 맥락을 보강한다.

출력:
- 이번 회차의 한 문장 핵심 프레임
- 독자가 관심 가질 이유
- 선택한 카드뉴스 각도
- 선택한 핵심 소재 3~5개
- 버린 소재와 이유
- 사실검증 결과와 남은 확인 필요 항목
- 위험한 표현 또는 확인 필요 표현
- 국내외 맥락 차이
- 독자에게 바로 와닿는 비유 또는 구조
- 6장 흐름 초안

좋은 프레임 예:
- “요즘 뜨는 ○○가 아니라, 작은 브랜드가 당장 고객을 잃는 지점을 보여준다.”
- “업계 뉴스처럼 보이는 이슈를 소비자 행동 변화로 바꿔 설명한다.”
- “큰 기업 사례를 그대로 따라 하기보다, 1인 사업자가 적용할 수 있는 최소 단위로 쪼갠다.”
- “해외에서 먼저 뜬 흐름이 국내에서는 어떤 형태로 바뀌어 들어오는지 보여준다.”
- “좋아 보이는 트렌드의 기회와 리스크를 한 장표 안에서 같이 보여준다.”

금지:
- 모든 리서치 결과를 다 넣으려 하지 않는다.
- 키워드 나열식 구성으로 가지 않는다.
- 출처가 약한 이야기를 흥미롭다는 이유만으로 핵심 메시지로 삼지 않는다.
- 독자가 바로 이해하기 어려운 업계 내부자용 내용으로 흐르지 않는다.

### Phase 3: Writing Outline

담당: `writing-outline`

역할:
- 확정된 메시지를 바탕으로 카드 6장의 역할과 한 줄 훅을 만든다.
- 아직 최종 본문을 길게 쓰지 않는다.
- 디자인이 잡을 수 있도록 카드별 정보 밀도와 시각 장치 후보를 같이 제안한다.

출력:
- 카드 1~6의 역할
- 카드별 핵심 문장
- 카드별 보조 문장
- 카드별 추천 시각 장치
- 반드시 들어가야 할 사실/주의 문구

권장 흐름:
1. Hook
2. Problem
3. Map
4. Guardrail
5. Before/After
6. Checklist

### Phase 4: Design

담당: `design`

역할:
- `writing-outline`과 `DESIGN.md`를 기준으로 디자인 시스템을 확정한다.
- 카드별 레이아웃, 컬러, 폰트, 도표 방식, 사진 사용 여부를 정한다.
- `image`와 `packaging`이 그대로 따라 할 수 있는 브리프를 만든다.

출력:
- 전체 톤앤매너 한 줄
- 컬러 팔레트
- 폰트 규칙
- 카드별 레이아웃 타입
- 카드별 시각 장치
- 반복 컴포넌트 규칙
- 사진이 필요한 카드와 사진 방향
- 금지 스타일
- 최종 QA 체크리스트

디자인 기준:
- 컴팩트하고 트렌디한 마케팅 스튜디오 인스타 캐러셀 느낌
- 검정, 크림, 오렌지 기반
- 큰 제목, 짧은 문장, 도표/리포트/체크리스트 중심
- 귀여운 툰형, 손그림, 파스텔 노트 템플릿은 사용하지 않는다.

### Phase 5: Writing + Image

담당: `writing`, `image`

진행:
- `writing`과 `image`는 `design` 확정 뒤 병렬 진행한다.
- 둘 다 `design` 브리프를 따른다.

`writing` 역할:
- 디자인 공간 제약에 맞춰 최종 카드 문구를 작성한다.
- 카드뉴스 성격에 맞게 짧고 크게 읽히는 문장으로 쓴다.
- 한 카드에는 하나의 핵심만 둔다.
- 모든 문구는 한국어로 작성한다.

`writing` 출력:
- 카드별 제목
- 카드별 보조 문장
- 도표/칩/체크리스트 문구
- 푸터 주의 문구
- 출처/확인 필요 문구

`image` 역할:
- `design`이 지정한 사진 또는 비주얼 소재를 만든다.
- 이미지 자체가 메시지를 설명해야 한다.
- 디자인 톤과 맞지 않는 귀여운 삽화, 손그림, 과한 캐릭터는 만들지 않는다.

`image` 출력:
- 필요한 PNG/JPG 소재
- 이미지 사용 위치
- 생성 프롬프트 또는 제작 설명

### Phase 6: Packaging

담당: `packaging`

역할:
- HTML/CSS를 구현한다.
- `DESIGN.md`와 `design` 브리프를 따른다.
- PNG 6장과 PDF를 출력한다.
- 최종 QA를 수행한다.

출력:
- `output/ai-weekly-cardnews.html`
- `output/cards/ai-weekly-card-01.png` ~ `output/cards/ai-weekly-card-06.png`
- `output/ai-weekly-cardnews.pdf`
- `output/ai-weekly-cardnews-render-check.png`
- `output/packaging-summary.md`

QA:
- 6장이 한 세트처럼 보이는가?
- 첫 장만 봐도 넘겨볼 이유가 있는가?
- 카드별 핵심이 한눈에 보이는가?
- 텍스트가 카드 밖으로 나가지 않는가?
- 도표/칩/체크리스트가 이해를 돕는가?
- PNG와 PDF가 최신 HTML에서 생성됐는가?

### Phase 7: Design Review

담당: `design-review`

역할:
- 패키징이 끝난 실제 PNG/PDF/HTML을 기준으로 디자인 완성도와 메시지 전달력을 검수한다.
- `design-review`는 리서치, 본문 작성, 이미지 제작, 패키징을 직접 하지 않고 검수 의견만 낸다.
- 카드별 핵심 메시지가 숫자만 남지 않고 “무엇을 말하려는지” 보이는지 확인한다.
- 폰트 굵기, 숫자 표기, 정렬, 여백, 로고, 푸터, 도표 라벨, 모바일 가독성을 확인한다.
- 문제가 있으면 P0/P1/P2 우선순위로 표시하고, 수정 방향을 구체적으로 제안한다.
- 검수 통과 전에는 최종 완료로 보고하지 않는다.

출력:
- `output/design-review-*.md`
- 카드별 P0/P1/P2 이슈
- 수정해야 할 문구/레이아웃/도표 요소
- 통과 여부

검수 기준:
- 한 장만 봐도 핵심 문장이 이해되는가?
- 수치가 단독으로 떠 있지 않고 해석 문장과 붙어 있는가?
- 제목, 보조문, 도표, 푸터의 폰트 체계가 일관적인가?
- 카드별 정보 밀도가 비슷하고 시선 흐름이 자연스러운가?
- 장식이 메시지를 방해하지 않는가?
- 1080px 폭 모바일 화면에서 작은 글자가 너무 많지 않은가?

## 보고 규칙

각 에이전트는 맡은 일이 끝났을 때 한 번만 보고한다.

보고 형식:

```text
결론:
1. ...
2. ...
3. ...

근거:
- ...
- ...

다음 단계:
- ...
```

예외적으로 즉시 보고해야 하는 경우:
- 막혀서 더 못 갈 때
- 사실 판단이 필요할 때
- 사용자 확인이 필요할 때
- 이전 단계 산출물과 충돌할 때

## 하위 에이전트 표시 규칙

`team-lead`는 하위 에이전트를 사용할 때 사용자가 현재 진행 상황을 직관적으로 볼 수 있게 표시한다.

기본 표기는 다음 형식을 따른다.

```text
[research / Rawls] 진행 중: 국내 2030 주식 자료 확인
[overseas / Mendel] 완료: 해외 비교 축 정리
[design / Nietzsche] 오류: 컨텍스트 초과 → team-lead 대체
```

표시 원칙:
- 역할명과 닉네임을 함께 표시한다.
- 상태는 `대기`, `진행 중`, `완료`, `오류`, `대체` 중 하나로 간단히 쓴다.
- 에이전트 ID는 사용자가 요청하거나 디버깅에 필요할 때만 노출한다.
- 에이전트 결과를 취합할 때는 “누가 무엇을 맡았고 어디까지 완료했는지”를 먼저 요약한다.
- 하위 에이전트가 실패하면 실패 사유와 `team-lead`가 대체한 범위를 최종 보고에 명시한다.

## Team Lead 운영 규칙

`team-lead`는 다음을 수행한다.

- Phase 전환 판단
- 에이전트 보고 취합
- 다음 에이전트 지시
- 사용자에게 요약 보고
- 오류 발생 시 우회 방향 결정
- 최종 산출물 검수

`team-lead`는 다음을 하지 않는다.

- 직접 리서치 본문을 작성하지 않는다.
- 직접 최종 카피를 쓰지 않는다.
- 직접 이미지를 제작하지 않는다.
- 필요 이상으로 에이전트에게 진행 상황을 묻지 않는다.

단, 현재 세션에서 도구 제약이나 에이전트 부재로 진행이 막히면 `team-lead`가 임시로 역할을 대체할 수 있다. 이 경우 최종 보고에 어떤 역할을 대체했는지 명시한다.

## 사용자 확인 정책

기본은 게이트별 확인이다.

```text
Phase 완료 → 요약 보고 → “다음 단계로 진행할까요?” → 사용자 승인 후 진행
```

단, 사용자가 “게이트까지 쭉 진행”, “오류 생기면 멈추지 말고 진행”처럼 자동 진행을 명시하면 승인 대기를 생략하고 계속 진행한다.

## 저장 위치

- 전체 운영 기준: `AGENTS.md`
- 디자인 기준: `DESIGN.md`
- 최종 산출물: `output/`
- 카드 PNG: `output/cards/`
- 이미지 소재: `output/photos/` 또는 `output/images/`
- 요약 보고: `output/packaging-summary.md`

에이전트 관련 규칙은 기본적으로 이 `AGENTS.md` 하나에 저장한다. 별도 파일은 프로젝트가 커져서 유지보수가 어려워질 때만 만든다.

## 브랜드 로고 규칙

- EARLYSHINE 로고 원본은 `assets/logo/main_a.png`에 저장한다.
- 카드뉴스 출력용 투명 배경 로고는 `assets/logo/main_a_transparent.png`를 사용한다.
- `design`은 모든 카드 오른쪽 상단에 실제 EARLYSHINE 로고가 들어가는 레이아웃을 전제로 잡는다.
- `packaging`은 텍스트 `EARLYSHINE` 대체 표기가 아니라 실제 로고 이미지를 삽입한다.
- `packaging`은 본문 카드 뒤에 로고 엔딩 카드를 1장 추가한다.
- 엔딩 카드는 중앙 로고 중심으로 구성하고, 카드뉴스 총 장수와 PNG/PDF 출력에도 포함한다.
